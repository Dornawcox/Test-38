This snippet works identically on any platform. Sites only need the Pages base URL:

const REGISTRY = 'https://YOUR-PAGES-URL/data'; // set per deployment
const MY_COMMUNITY = 'farmhack';                 // set per site
async function loadCards(type) {
  const r = await fetch(`${REGISTRY}/${type}.json`);
  return (await r.json()).filter(c => c.communities.includes(MY_COMMUNITY));
}

Help & Documentation

How the Registry Works

The Commons Card Registry is a single GitHub repository containing JSON data files. Each file holds an array of cards (tools, projects, events, etc.). Community websites fetch these files and filter by their community tags to display relevant content.

No one needs to know Git. The admin panel reads and writes JSON through the GitHub API. Changes appear on all community sites within ~1 minute (the time it takes GitHub Pages to rebuild).

Creating a New Card

  1. Click "New Card" and select the card type
  2. Fill in the required fields (title, description, tags, communities)
  3. Click "Save" — if you're a steward, it commits directly; if you're a contributor, it creates a pull request for review
  4. The card appears on all community sites that match its community tags

Making a Card Appear on Multiple Sites

Add community IDs to the card's communities array. For example, a tool tagged ["farmhack", "giaa", "openteam"] will appear on all three sites. Only stewards of a community can add/remove that community's tag.

The "Browse All Communities" Toggle

Every community site includes a toggle that lets visitors see cards from all communities, not just their own. This promotes cross-pollination while keeping the default view curated and focused.

API Reference: GitHub Contents API

The admin panel uses these GitHub API endpoints:

Read: GET /repos/{owner}/{repo}/contents/data/{file}.json

Write: PUT /repos/{owner}/{repo}/contents/data/{file}.json (include sha from read, content as base64, message as commit message)

Create PR: POST /repos/{owner}/{repo}/pulls (first create a branch with POST /repos/{owner}/{repo}/git/refs)

Auth: Include header Authorization: Bearer {token} on all requests.

ETA Registry

Dedicated registry for Ethical Technology Assessment reports. Each entry is a scored assessment tied to a specific tool, dataset, or project card in the commons registry.

Open ETA Tool ↗

ETA Registry — how it relates to other registries

Each ETA assessment references an entry in another registry (tool, dataset, or project) via assessed_tool_id. The assessment score is then reflected as a badge on the referenced card in that registry. Assessments are versioned and re-runnable — a tool can have multiple assessments over time.

Dependencies: ETA entries link to tools, datasets, and projects registries by ID. The CI build validates that assessed_tool_id exists in one of those registries before publishing. The score is denormalised back into the referenced card JSON at build time.

Contribution Form

Configure and deploy the live contribution form that lets community members propose new registry entries.

How the live contribution form works on GitLab

When a community member submits the form, their browser builds a structured JSON entry and opens a pre-filled GitLab new-file page — or sends it as a GitLab Issue if they have no git account. No server required. All processing happens in the browser.

Three submission paths, for different contributor types:

  1. Git-native: Opens gitlab.com/ORG/REPO/-/new/main/?file_name=registry/…&content=… — contributor commits directly.
  2. Issue-based: Opens a pre-filled GitLab Issue with JSON in the body — steward reviews and commits manually.
  3. Email fallback: Generates a mailto: link with the JSON attached as text — for contributors with no git account.

Setup: 4 steps to a live contribution form

  1. Configure your registry repo below (org/repo path on GitLab).
  2. Choose your submission mode: direct commit, Issue, or email.
  3. Copy the embed snippet and paste it into any HTML page — it runs entirely in the browser, no backend needed.
  4. Optional: add a GitLab CI webhook that auto-labels new Issues tagged registry-submission and assigns them to stewards.
