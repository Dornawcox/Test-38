Setup & Configuration

⚙ Step 1: Connect to GitHub Repository

The admin panel reads and writes card data via the GitHub API. You need a Personal Access Token (PAT) with repo scope, or connect via OAuth (for production).

  1. Go to GitHub Settings → Developer Settings → Personal Access Tokens
  2. Click Generate new token (classic)
  3. Name it commons-registry-admin
  4. Select scope: repo (full control of private repositories)
  5. Copy the token and paste it below
Stored in browser session only. Cleared on tab close.
The GitHub repo where registry data lives.

🔒 Step 2: Set Up Branch Protection

Protect the main branch so that contributors must submit pull requests, and only stewards can approve.

  1. Go to Repository Settings → Branches → Add branch protection rule
  2. Branch name pattern: main
  3. Enable: Require pull request reviews before merging (1 approval)
  4. Enable: Require review from Code Owners
  5. Enable: Require status checks to pass (select the validate workflow)
  6. Click Create

Note: Registry Admins can bypass protection for urgent fixes via "Allow specified actors to bypass."

👥 Step 3: Create GitHub Teams

Teams map to community steward roles. Each team gets write access to the registry repo.

  1. Go to your GitHub Organization → Teams → New Team
  2. Create teams: registry-admins, farmhack-stewards, openteam-stewards, giaa-stewards, concordance-stewards
  3. Add members to each team
  4. Grant each team Write access to the commons-registry repo

📄 Step 4: Add CODEOWNERS File

Create .github/CODEOWNERS in the repo root. This controls who must review changes to each data file:

# Commons Registry CODEOWNERS
# Changes to these files require approval from the listed teams

data/tools.json          @org/farmhack-stewards @org/openteam-stewards
data/projects.json       @org/openteam-stewards @org/giaa-stewards
data/events.json         @org/farmhack-stewards @org/giaa-stewards @org/openteam-stewards
data/library.json        @org/concordance-stewards @org/openteam-stewards
data/datasets.json       @org/concordance-stewards @org/openteam-stewards

schemas/*                @org/registry-admins

✅ Step 5: Add Validation Workflow

Create .github/workflows/validate.yml to automatically check JSON validity on every push and PR:

name: Validate Registry Data
on: [push, pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate JSON
        run: |
          for f in data/*.json; do
            python -m json.tool "$f" > /dev/null || exit 1
            echo "✓ $f is valid JSON"
          done
      - name: Check required fields
        run: |
          python3 -c "
          import json, sys
          for fname in ['tools','projects','events','library','datasets']:
            with open(f'data/{fname}.json') as f:
              cards = json.load(f)
            for c in cards:
              for field in ['id','type','title','communities','tags']:
                if field not in c:
                  print(f'ERROR: {fname}.json card missing {field}: {c.get(\"id\",\"unknown\")}')
                  sys.exit(1)
          print('✓ All required fields present')
          "

🔗 Step 6: Enable GitHub Pages

This makes the JSON data files accessible as static URLs that any community site can fetch.

  1. Go to Repository Settings → Pages
  2. Source: Deploy from a branch
  3. Branch: main, folder: / (root)
  4. Click Save

After ~1 minute, your data is live at:
https://dornawcox.github.io/commons-registry/data/tools.json

Community sites fetch this URL to display cards. No API keys needed for reading — it's public.

🌐 Step 7: Add Community Sites (Consumer Setup)

Each community website includes a small JavaScript snippet to fetch and display cards. Copy this into any GitHub Pages site:

const REGISTRY = 'https://dornawcox.github.io/commons-registry/data';
const MY_COMMUNITY = 'farmhack'; // change per site

async function loadCards(type) {
  const res = await fetch(`${REGISTRY}/${type}.json`);
  const all = await res.json();
  return all.filter(c => c.communities.includes(MY_COMMUNITY));
}

// Load tools, render into page
loadCards('tools').then(tools => {
  const grid = document.getElementById('tool-grid');
  tools.forEach(t => {
    grid.innerHTML += `
      <div class="card">
        <h3>${t.title}</h3>
        <p>${t.description}</p>
        <div>${t.tags.map(tag =>
          '<span class="tag">'+tag+'</span>').join('')}
        </div>
      </div>`;
  });
});

Help & Documentation

How the Registry Works

The Commons Card Registry is a single GitHub repository containing JSON data files. Each file holds an array of cards (tools, projects, events, etc.). Community websites fetch these files and filter by their community tags to display relevant content.

No one needs to know Git. The admin panel reads and writes JSON through the GitHub API. Changes appear on all community sites within ~1 minute (the time it takes GitHub Pages to rebuild).

Creating a New Card

  1. Click "New Card" and select the card type
  2. Fill in the required fields (title, description, tags, communities)
  3. Click "Save" — if you're a steward, it commits directly; if you're a contributor, it creates a pull request for review
  4. The card appears on all community sites that match its community tags

Making a Card Appear on Multiple Sites

Add community IDs to the card's communities array. For example, a tool tagged ["farmhack", "giaa", "openteam"] will appear on all three sites. Only stewards of a community can add/remove that community's tag.

The "Browse All Communities" Toggle

Every community site includes a toggle that lets visitors see cards from all communities, not just their own. This promotes cross-pollination while keeping the default view curated and focused.

API Reference: GitHub Contents API

The admin panel uses these GitHub API endpoints:

Read: GET /repos/{owner}/{repo}/contents/data/{file}.json

Write: PUT /repos/{owner}/{repo}/contents/data/{file}.json (include sha from read, content as base64, message as commit message)

Create PR: POST /repos/{owner}/{repo}/pulls (first create a branch with POST /repos/{owner}/{repo}/git/refs)

Auth: Include header Authorization: Bearer {token} on all requests.

Project Cards

Project cards follow the same pattern as tool cards. Filtered by project_type: collabathon, pilot, working-group, research.

Event Cards

Event cards include date_start, date_end, location, virtual flag, and registration_url. Shared across community calendars.

Library Cards

Library cards cover books, papers, frameworks, reports, videos, and datasets. Filtered by media_type and topics.

Dataset Cards

Dataset cards include format, spatial/temporal extent, access tier, provenance, and consent model. Critical for data sovereignty.

Tags & Taxonomy

Controlled vocabulary for card tags. Categories: hardware, software, protocol, design, event-type, region, license, etc.
